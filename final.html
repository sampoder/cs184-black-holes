Abstract

Our goal with this project is to build an interactive real-time black hole visualization and simulation that models gravitational lensing around a Schwarzschild-style black hole using numerical ray tracing. Standard ray tracing relies on the optical assumption that light moves and is affected in straight lines. But, the laws of relativity state that light moves differently around objects with large masses (black holes), so rendering such objects accurately requires changing the ray tracer. To do so, rays are integrated through curved spacetime to render the black hole shadow, surrounding starfield, and a physically inspired accretion disk with relativistic shading effects such as Doppler boosting and gravitational redshift. Users can experience how matter and light behave near an extreme gravitational field by creating orbiting objects whose trajectories are affected by the black hole. Combined with interactive camera controls, the project serves as both a visual demonstration of black hole physics and a computational graphics study in real-time relativistic rendering.

Technical approach

Starting Point: We used no starter code and built the repo/project from scratch.

Display: Our simulation renders an 800×800 image of a black hole on the CPU using a custom ray tracer that numerically integrates photon geodesics through Schwarzschild spacetime.  We used OpenGL as a display layer to show the finished pixel buffer in the window and to draw the IMGui overlay. The CPU computes the black hole image through ray tracing, stores the result to displayPixels, and draws that pixel buffer to OpenGL using drawFrame().

Main Tracing: Each pixel is rendered by marching a ray through curved spacetime, bending its direction at every step according to a = -(3/2) * rs * h^2 / r^5 * pos. This bending is what makes the stars behind the black hole appear displaced and sometimes duplicated into the rings. h^2 = |pos_0 x dir_0|^2 captures how obliquely the ray passes the black hole and controls how sharply it curves; it is computed once at the start of each ray and held constant. The integrator uses a symplectic (kick-drift) Euler update to keep ray paths stable over thousands of steps. Adaptive step sizing is used to prevent excessive computation (mostly where the image actually changes). It is set to a fixed 0.05 near the photon sphere where bending is sharpest and 0.08 in empty space where rays travel nearly straight. Hashing escaped ray directions (by choosing the brightness and color tint from the hash bits) onto an angular grid with ~2% star density gives the background stable star pattern that moves smoothly.

Extra Effects: The disk itself comes from layering several independently tunable effects. The base color is a three-band ramp which is controlled at runtime by the ImGui RGB sliders. The material on top of this rotates at Keplerian speed omega(r) = K / r^(3/2) (K = 30), so the disk frame angle at render time is phi_0 = phi - Kt / (r * sqrt(r)), which is cheaper than a pow() call. Two sinusoidal patterns in this rotating frame break the disk into drifting bright clumps that orbit faster near the center, giving the impression of turbulent infalling material. Finally, a Doppler factor = 0.6 + 0.4 * sin(phi + 1.0) scales the three RGB channels, brightening the side of the disk rotating toward the camera and dimming the receding side. This last part was most needed to show the relativistic effect of the black hole. Users can also spawn glowing particles with the spacebar that follow the equation a = -(pos / r^3) * (1 + 3h^2 / r^2), where the correction term 3h^2 / r^2 is what causes particles that pass too close to fall into the event horizon rather than returning.

Variation from References and Unique Decisions: Compared to Luminet (1979), which used semi-analytical 2D ray tracing with proper radiative transfer, we do full 3D numerical ray marching to support different camera angles. Our Doppler effect is a sinusoidal approximation rather than the proper relativistic factor from Penrose & Floyd (1971), and gravitational redshift is folded into the color ramp rather than computed as a wavelength shift. Unlike GPU-based implementations used in production (James et al. 2015), we made ours run entirely on the CPU with a custom thread pool, so we used a different rendering system (with atomic buffer swapping) to make it usable.

Problems and Solutions:

  • Even though the project focused on higher-level ray-tracing, lower-level knowledge from 61C/184 implementations greatly helped make the simulation usable.

We were initially rendering line by line, but processing pixels in 16×16 tiles kept the working set for traceRay in L2/L3 cache for the duration of each tile, reducing memory latency on the pixel buffer. We used double-buffering to eliminate tearing without needing to use per-pixel locking.

  • The hardest challenge we faced was that none of us had backgrounds in numerical analysis, so we had to experiment with different techniques to see what worked best since many existing implementations were GPU-optimized (while we built a CPU one).

We had started with standard forward Euler to model the particle dynamics, but this increased the total energy and led to simulation errors building up (eg. particles spiraling outward) after ~30 seconds. Switching to a symplectic integrator (kick before drift) kept orbit shapes stable for much longer (ie. we observed no issues even after >20 minutes running).

  • Switching to adaptive step sizes was necessary for rendering black holes.

The area near the blackhole (where spacetime is bent) has extreme curvature while the areas at the edges of the screen have near zero curvature. Our initial approach used a fixed step that was small enough near the black hole, meaning that there was excessive computation in empty space. Optimally calculating the radii at which the effects needed to be rendered differently (eg. center, photon sphere, far away) helped improve image quality, particularly during transitions.

  • Using AI can help with cross-platform collaboration (Windows, Mac), particularly in this setup (C++ in Visual Studio).

All of us had previously done group projects (eg. 61B, 61C, 162) where different people had Windows/Mac/Linux and basic Git knowledge was sufficient for collaboration. However, since we built the directory from scratch, we ran into many issues making Windows implementations work on Mac and vice versa. Using AI to make any necessary changes after each pull request greatly helped reduce this workload.

Lessons Learned: We found that our 184 knowledge (largely Project 3) was invaluable for getting a functioning pixel buffer and a basic version of the black hole renderer ready. We didn’t realize how easily the math behind ray tracing was applicable to this problem. But, using just the standard techniques led to incredibly slow performance. We learned that mathematical and low-level programming knowledge was also necessary for this project. While we initially expected that the extra optimization would be easier than the initial rendering, we realized that we had to find/use more advanced techniques for our use case, specifically in numerical analysis and caching. We also learned that getting a universal version of the code up and running from the start is really helpful with preventing cross-platform collaboration issues in large-scale projects.
